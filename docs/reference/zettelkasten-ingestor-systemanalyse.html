A Kurzdiagnose des aktuellen App-Zustands

Was am aktuellen Stand schon gut ist

Der Code ist solide. Die Persistenz via localStorage mit strukturierten Defaults, die Normalisierungs-Funktionen für Notizobjekte und MOC-Einträge, das Debouncing, der Log-Stream, das Workspace-Import/Export — das sind alles Bausteine, die man in Phase 1 der Überarbeitung direkt weiterverwenden kann. Auch das Dateimodell für structuredNote (id, title, summary, basePath, slug, tags, type, flashcards) ist konzeptuell schon nah am Ziel.

Wiederverwenden

localStorage-Persistenz, Normalisierungs-Funktionen, File-Drop-Zone, Activity-Log, Workspace-Backup, die structuredNote-Objektstruktur, ZIP-Export via JSZip.

Wo die App vom Ziel abgedriftet ist

Das zentrale Problem: Die App ist auf eine einzelne große SOP ausgerichtet, aber das Ziel ist atomare Zerlegung in viele kleine Notizen. Alles andere folgt aus diesem Grundmissverhältnis.

  1. SOP als Endformat. Die gesamte Kette läuft auf eine Markdown-Datei zu. Der Denoise-Prompt ist auf eine SOP-Struktur (## Trigger / ## Root Cause / ## Flashcards) getrimmt. Das ist für Chat-Exporte aus Debugging-Sessions richtig — für Unterrichtsmaterial falsch. Die SOP ist ein Nebenprodukt, nicht das Hauptprodukt.
  2. Das Rohtext-Textarea als Hauptarbeitsfeld. Für kleine Snippets okay. Für Chat-Exporte mit 10.000+ Zeichen falsch — das passt nicht in einen Prompt, und der User weiß das. Datei-Attach + Standard-Prompt ist der reale Workflow.
  3. Denoising als fest verdrahtete erste Stufe. Denoising macht Sinn für verrauschte Chat-Logs. Für strukturiertes Unterrichtsmaterial ist es oft unnötig oder kontraproduktiv. Es sollte optional sein, nicht obligatorisch.
  4. JSON-Import als sichtbarer Nutzer-Workflow. Der Button "📥 JSON importieren" suggeriert, dass JSON ein Endnutzer-Format ist. Ist es nicht. JSON ist das interne Rückgabeformat der KI.
  5. Konfiguration zu prominent. API-Keys, Modell-Wahl, Temperatur, Obsidian-URL — diese Dinge braucht man einmal. Die ganze rechte Sidebar für Config zu reservieren verdrängt das Wesentliche.
  6. MOC-Bezug zu spät und zu versteckt. Die MOC taucht erst in Schritt 4 ("Generated Notes") auf, nach Denoising und SOP-Erstellung. Das ist konzeptuell falsch: die MOC muss vor der Analyse-Prompt-Generierung geladen sein, weil sie Kontext für den Prompt ist.
  7. Keine sichtbare Pfad/Zweig-Bearbeitung. Das Kernfeature — Zweige prüfen, umbenennen, verschieben — gibt es in der UI nicht. Die structuredNotes-Felder (basePath, slug) sind im Datenmodell vorhanden, aber in der UI nicht wirklich editierbar.
  8. Flashcard-Felder als Textareas. Zwei Textareas (flashFront / flashBack) sind ein Einzel-Karten-Interface. Das Anki-Zielformat (3 CSVs: MC, Tapping, Cloze) und das bereits gut definierte Template werden vom UI-Konzept nicht abgebildet.
Ballast / falsch priorisiert

SOP-Prompt als Hauptpfad · Rohtext-Textarea als Primäreingabe · JSON-Import-Button als Nutzer-Workflow · große Config-Sidebar · API-Modus vs. Manuell-Toggle als primäre UX-Entscheidung · einzelne Flashcard-Front/Back-Felder.

B Klare Produktdefinition

Was diese App ist

KI-gestützter Zettelkasten-Ingestor für Unterrichtsnotizen, Chat-Exporte und Lernmaterial.

Die App nimmt rohes Lernmaterial, erzeugt daraus strukturierte Analyse-Prompts, interpretiert die strukturierten KI-Antworten und exportiert atomare Markdown-Notizen in einer baumförmigen Struktur — kompatibel mit Obsidian Structured Tree und MOC-basiertem Zettelkasten.

Sie ist ein Prompt-Generator + Struktur-Editor + Export-Engine. Die KI sitzt außerhalb der App (Claude / ChatGPT). Die App ist die Klammer darum.

Was diese App nicht ist

  • Kein SOP-Builder (SOPs können Nebenprodukt sein, sind aber nicht Hauptzweck)
  • Kein generisches ETL-Tool
  • Kein Blackbox-KI-Wrapper (keine direkte API-Integration als Hauptpfad)
  • Kein Anki-Generator (Anki-CSV ist ein optionaler Nebenoutput)
  • Kein Multi-File-Projekt und kein Build-Step-Tool

C Ziel-Workflow

1
Nutzer
Material auswählen
Dateien droppen (MD, TXT, JSON-Export, HTML) oder Text einfügen. App liest Dateien ein und zeigt Zusammenfassung (Dateiname, Zeichenzahl, erkannter Typ).
2
Nutzer
Bestehende MOC laden (optional)
MOC-Markdown-Datei einlesen. App parsed Wikilinks + Einzeiler-Summaries und baut intern MocIndex[] auf. Anzeige: wieviele MOC-Einträge geladen, welche Tiefe, welche Top-Zweige. Wenn keine MOC: Bootstrap-Modus aktivieren.
3
App · intern
Analyse-Prompt erzeugen
App assembliert Prompt aus: Systeminstruktion + MOC-Kontext (wenn vorhanden) + Rückgabeformat-Spezifikation. Der Prompt erklärt der KI, dass sie das Material in atomare Notizen zerlegen und als strukturiertes JSON zurückgeben soll.
4
Nutzer
Prompt kopieren + extern an KI senden
Prompt in Zwischenablage. Nutzer öffnet Claude / ChatGPT, fügt Prompt ein, hängt Dateien an (weil Dateien zu groß für Prompt-Embedding sind). KI antwortet mit strukturiertem JSON.
5
Nutzer
KI-Ergebnis einfügen
JSON-Antwort der KI in die App einfügen. App validiert Struktur und zeigt Fehlermeldung wenn ungültig.
6
App · intern
Notizen parsen + gegen MOC einordnen
App liest notes[] aus dem JSON. Wenn MOC vorhanden: Pfad-Vorschläge aus KI mit MOC-Einträgen abgleichen, Konflikte markieren. Wenn Bootstrap: vorgeschlagene Struktur direkt übernehmen.
7
Nutzer
Pfade / Zweige prüfen und modifizieren
Jede atomare Notiz ist editierbar: Pfad (Struktur-ID), title, desc, tags, type. Zweige lassen sich umbenennen (wirkt auf alle Notizen im Zweig). Reihenfolge sortierbar. Neue Notizen manuell ergänzen.
8
Nutzer
Export auslösen
Markdown-Dateien (eine pro Notiz, Dateiname = Struktur-ID), MOC.md, optional Anki-CSVs (MC, Tapping, Cloze), optional ZIP-Bundle.
+
Später · Refactor-Ebene
Struktur-Review-Prompt erzeugen
App serialisiert bestehende MOC + Notizstruktur, erzeugt einen Refactor-Analyse-Prompt. KI gibt Umstrukturierungsvorschläge zurück. App stellt diese als einzeln bestätigbare Änderungen dar.

E Ziel-UI / Ziel-Informationsarchitektur

Die App sollte 5 klar getrennte, linear aufeinander aufbauende Hauptbereiche haben. Kein komplexes Sidebar-Layout. Ein Hauptpfad mit deutlichem Fortschritt.

Hauptbereiche (Ziel)

  1. Material — Datei-Drop, Datei-Liste, ggf. Zwischenablage-Paste. Kein großes Rohtext-Textarea mehr als Primärelement. KPI: Dateien, erkannter Typ, Zeichenzahl gesamt.
  2. Kontext (MOC) — MOC-Datei laden. Wenn geladen: Top-Zweige anzeigen, Anzahl Einträge, Tiefe. Wenn nicht geladen: deutlicher Hinweis "Bootstrap-Modus — KI schlägt erste Struktur vor". Wichtig: MOC-Kontext ist Input für den Prompt, deshalb vor der Prompt-Generierung.
  3. Analyse-Prompt — Erzeugter Prompt anzeigen. Copy-Button groß. Rückgabefeld für KI-JSON. Validierungsstatus. Kein Toggle "API vs. Manuell" als Hauptentscheidung — manuell ist der Default.
  4. Notizen-Editor — Die generierten atomaren Notizen als editierbare Karten. Jede Karte zeigt: path (editierbar), title (editierbar), desc (editierbar), type, tags, body (expandierbar). Zweig-Umbenennung: ein Edit-Feld für den Präfix wirkt auf alle Karten im Zweig. Konflikt-Warnung wenn MOC-Abgleich nicht eindeutig.
  5. Export — Markdown-Dateien (ZIP), MOC.md, optional Anki-CSVs (3 Dateien). Klare Checkliste was exportiert wird.

Sidebar (Ziel)

Schlanke Config-Sidebar mit Collapse-Funktion: Obsidian-URL, Pfad-Präfix, Anki-Deck. API-Keys nur wenn API-Modus gewünscht. Nicht mehr der dominante rechte Block.

F Prompt-Weitergabe-Architektur

Die App ist ein Prompt-Generator, keine KI-Blackbox. Es gibt genau drei Prompt-Typen:

Prompt 1 · Analyse & Zerlegung (Haupt-Prompt)

Erzeugt aus: Systeminstruktion + MOC-Kontext + Rückgabeformat-Spezifikation. Das Material wird nicht in den Prompt eingebettet — es wird als Datei-Anhang an die KI geschickt.

  • Instruiert die KI, das Material in atomare Notizen zu zerlegen
  • Enthält die MOC als Kontext (wenn vorhanden) — als komprimierte Baumdarstellung
  • Gibt das JSON-Rückgabeformat vor (Struktur, Pflichtfelder, Beispiel)
  • Im Bootstrap-Modus: KI soll selbst eine erste Baumstruktur vorschlagen

Prompt 2 · Anki-Karten (Nebenoutput)

Erzeugt aus: exportierten atomaren Notizen + Anki-Template-Spezifikation. Nutzt das bereits gut definierte anki_csv_prompt.md als Basis.

  • Pro Fach / pro Notiz-Cluster einmal ausführen
  • Gibt 3 CSVs vor (MC, Tapping, Cloze)
  • Material: bereits bereinigte atomare Notizen, nicht das Rohmaterial

Prompt 3 · Refactor-Analyse (Später)

Erzeugt aus: aktuelle MOC-Serialisierung + Notizstruktur. KI gibt strukturierte Umstrukturierungsvorschläge zurück (JSON: Typ der Änderung, betroffene Pfade, Begründung).

Rolle von Denoising

Denoising ist optional und kontextabhängig. Für verrauschte Debug-Chats: sinnvoll. Für strukturiertes Unterrichtsmaterial: oft überflüssig. Es sollte als optionaler Pre-Processing-Schritt existieren, den der Nutzer bewusst aktiviert — nicht als feste erste Stufe der Kette.

Wenn aktiviert: eigener Prompt, eigene Copy-Paste-Runde, dann Ergebnis als Input für Prompt 1.

Strukturierte Rückgabeformate

Nur an zwei Stellen sinnvoll: nach Prompt 1 (→ Notizen-JSON) und nach Prompt 3 (→ Refactor-Vorschläge-JSON). Anki-CSVs sind kein JSON sondern direkt nutzbare CSV-Dateien.

G Vorschlag für das KI-Rückgabeformat

Das Format muss einfach genug sein, dass Claude / ChatGPT es zuverlässig produzieren, und reich genug, dass die App daraus alle Exports ableiten kann.

Warum kein Markdown als Rückgabeformat?

Weil die App aus einem KI-Output mehrere Notizen mit je eigenen Metadaten, Pfaden und Flashcards extrahieren muss. Das ist mit Markdown-Parsing fragil. JSON ist eindeutig parsebar, validierbar und direkt in Exports umwandelbar.

Was die App daraus erzeugt

  • Markdown-Datei pro Notiz: Dateiname = path + ".md". Frontmatter aus title, desc, type, tags. Body aus body-Feld.
  • MOC.md: Aus allen desc-Feldern + Wikilinks auf alle paths.
  • Anki-CSVs (3 Stück): Aus allen flashcards[] nach cardType gruppiert.

Frontmatter-Ausgabeformat (pro .md-Datei)

I Single-File-Realisierbarkeit

Die Zielarchitektur ist als Single-File-HTML realistisch umsetzbar. Die Komplexität ist beherrschbar, weil:

  1. Kein Reaktivitäts-Framework nötig. Die App hat einen linearen Workflow. Vanilla-JS mit klarer render()-Funktion pro Bereich reicht. Der aktuelle Code beweist das bereits für 3.000 Zeilen.
  2. Saubere Trennung von Logik und UI. Alles unter // --- DATA --- (Normalisierung, Parsing, Export) hat keine DOM-Abhängigkeiten. Das lässt sich gut in Funktionsblöcke im <script>-Tag organisieren.
  3. Externer KI-Aufruf nimmt Komplexität raus. Weil die KI extern läuft, gibt es keine Async-Chains, keine Retry-Logik, keine Streaming-Implementierung in der App. Das vereinfacht.

Wie man Komplexität begrenzt

ProblemLösung
Zu viele Features im Scope Phasen 1–4 zuerst fertig machen, bevor Denoising oder Refactor-Ebene gebaut wird
Langer <script>-Block Funktionale Blöcke mit Kommentar-Headern: DATA / PARSE / RENDER / EVENTS / EXPORT. Keine Klassen nötig.
UI-Komplexität im Editor Karten per innerHTML-Template rendern. Kein Virtual DOM. Event-Delegation auf Container.
localStorage-Größe Body-Felder kürzen wenn > 5kB pro Notiz. Blob-Inhalte nie speichern.
Sichtbare Komplexität Config-Bereich kollapsiert. Denoising-Bereich kollapsiert wenn nicht aktiviert. Schritte 1–5 als sichtbare Progress-Indikatoren.
Empfehlung

Neues Grundgerüst von Grund auf schreiben (Phase 1) statt den aktuellen Code zu patchen. Dann gezielt die wiederverwertbaren Funktionen aus dem alten Code hinüberkopieren. Der aktuelle Code ist nicht schlecht — aber die Grundstruktur hat sich zu weit vom Ziel-Workflow entfernt, um sie per Patch zu retten.
